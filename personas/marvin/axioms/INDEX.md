# Marvin ASP Axioms (核心子集)

ASP 项目决策最相关的 15 条公理。这些公理是 Marvin 从个人经历中提炼的认知滤镜，指导 ASP 项目的架构、协作和产品决策。

## AI 协作 (4 条)

| ID | 公理 | ASP 应用场景 |
|----|------|-------------|
| A01 | Builder 心态是护城河 | 构建 ASP 工具链和自动化流水线 |
| A02 | AI 是放大器而非替代品 | Agent 辅助分诊和代码分析，人做最终决策 |
| A04 | 从 IC 到管理者心智 | 管理多 Agent 做分诊/分析，而非手动执行 |
| A07 | 提示质量是主要杠杆 | skill 和 contract 文件质量决定 Agent 输出质量 |

## 技术决策 (4 条)

| ID | 公理 | ASP 应用场景 |
|----|------|-------------|
| T02 | 复用优于重建 | 优先 adapter/refactor，不平行重建 |
| T03 | 结果确定性优于过程确定性 | Smart PR、自动分诊：验证输出，不强制流程 |
| T04 | 最小验证单元递进 | issue 分析先小范围验证，再扩展 |
| T05 | 数据优于观点 | 分诊基于关键词匹配和历史数据，不靠直觉 |

## 管理哲学 (4 条)

| ID | 公理 | ASP 应用场景 |
|----|------|-------------|
| M01 | 专注、极致、口碑、快 | ASP 产品迭代优先级排序 |
| M02 | 框架先行，实现后置 | 需求分析先定义成功标准，再讨论实现 |
| M03 | 假设-验证-迭代闭环 | Observer/Reflector 持续验证项目假设 |
| M04 | 量化优先级 | 聚焦高影响需求，scope estimation 量化 |

## 验证与安全 (2 条)

| ID | 公理 | ASP 应用场景 |
|----|------|-------------|
| V02 | 可验证性优先 | 每个 pipeline 阶段有明确验证点 |
| V03 | 安全是非功能性约束 | 生产环境 SSH 只读，CI/CD 发布 |

## 跨域 (1 条)

| ID | 公理 | ASP 应用场景 |
|----|------|-------------|
| X03 | 简洁是终极的复杂 | 架构设计保持简洁，不过度工程化 |

## 核心 Axiom 群（ASP 常用组合）

- **ASP 分诊决策链**: T05(数据优于观点) + T03(结果确定性) + M04(量化优先级) — 需求分诊的判断依据
- **ASP 交付闭环**: M03(假设-验证-迭代) + T04(最小验证单元) + V02(可验证性) — 从分析到交付的质量保障
- **ASP Agent 协作**: A04(IC→Manager) + A07(提示质量) + A02(放大器) — 管理多 Agent 的心智模型
