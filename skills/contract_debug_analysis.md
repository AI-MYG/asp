# Debug Analysis Contract

## 元数据

- **类型**: Contract
- **适用场景**: Team Lead 本地 Agent 对执行 issue 的分析 comment 输出格式

## Comment 结构

> **第 0 章「给需求方的话」**（Pipeline C 输出的分析报告新增）：报告最前面必须有一段大白话、零术语、3-5 句话的总结，写给非技术需求方看——说清「需求理解成什么 / 打算怎么改 / 改完什么效果 / 有无风险」，并以固定提示结尾：「✅ 确认无误后，请给本 issue 打上 `approved-to-execute` 标签，我就会开始改代码。」人类读这段确认需求，才进入 Pipeline D 改代码。

```markdown
## Analysis

**Surface**: {surface}
**Difficulty**: {difficulty}
**Estimated Scope**: {scope_description}

### Root Cause / Requirement Mapping

{requirement_to_code_mapping}

### Proposed Approach

{approach_with_file_paths_and_rationale}

### Test Strategy

{what_to_test_and_how}

### Risk / Dependencies

{cross_surface_dependencies_or_risks}

---
_Generated by ASP local agent. Central issue: AI-MYG/asp#{N}_
```

## 原则

- 每个 section 必须有实质内容，空 section 不如不写
- Root Cause 要映射到具体文件和行号（如已知）
- Proposed Approach 要说明"为什么这样做"，不只是"做什么"
- Risk 部分重点关注跨 surface 影响
