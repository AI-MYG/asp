## 1. KET 页面跳转链路

从 `ket_kingsley_page.dart`（主地图）出发，有 3 个入口：

```
KET Kingsley 主页 (/ket/kingsley)
│
├─ 1. 点击荷叶 → Unit Courses 课程列表 (/ket/unit-courses)
│  └─→ 点击课程卡片 → Resource Loader 资源加载 (/ket/resource-loader)
│     └─→ 拉取 game-index + 下载解压 + 写 pack 绑定
│        └─→ pushReplacement 到树桩分发页 (/ket/loading)
│           └─→ 用户点树桩 或 auto_fetch_pack=1 自动分发：
│              ├─ collect_it  → Collect It Main (/ket/collect-it-main)
│              │  └─→ 点元素触发 jump → Collect It 子包 (/ket/collect-it)
│              │     ├─→ 未全通关 → pop 回 main，main 洞变正常展示
│              │     └─→ 全通关    → 庆祝动画 → go /ket/loading
│              ├─ decode_it   → Decode It (/ket/decode-it)
│              │  └─→ 通关庆祝 → go /ket/loading?auto_fetch_pack=1&game_type_name=phrase_it
│              ├─ phrase_it   → Phrase It (/ket/phrase-it)
│              ├─ fill_it (game) → /ket/friday
│              └─ sight_word_chant → Sight Word Chant (/ket/sight-word-chant)
│
├─ 2. 左下 card_deck → 儿童端首页（pop 或 /child/home，与原 mini_home 一致）
├─ 3. 左下 week_0 → Week 0 任务页 (/ket/week-missions，始终可进)
├─ 4. 点击书架 → Card Library 卡片库 (/ket/card-library)
│  └─→ 返回按钮回主页
│
└─ 5. 点击金币 → Sight Word Chant (/ket/sight-word-chant)
   └─→ Video Player → Theme Missions
```

**关键事实**：历史文档提到过的 `/ket/course-games` 路由在代码里**不存在**；
所有"回到分发点"语义都用 `/ket/loading` 实现。

## 2. 目标与核心设计

- 统一六阶游戏链路的资源包获取，避免"接口有返回但页面读不到缓存"的问题。
- 以 `pack_id` 作为子页面唯一入参；collect-it 子包玩法进一步做到
  "本地命中直接用，否则自拉自下自缓存"。
- 形成闭环：`resource-loader → loading → 玩法页 → loading`；其中
  `collect-it-main ↔ collect-it` 形成内部小闭环，洞未全完成前不离开 main。

### 2.1 业务缓存键

```
<courseUnitId 或 "all">@day-<courseDay 或 1>@<normalizedGameType>
```

- 实现：`buildKetGamePackCacheKey(...)`
  （`lib/ui/ket/config/ket_game_flow_config.dart`）。
- 规范化规则：
  - `gameTypeName` 走 `normalizeKetGameTypeName()`：支持 `-`/`_`，
    别名 `fill_it → order_it`；`game` 保留为独立类型；非法值回落 `collect_it`。
  - `courseUnitId` 空值写 `all`。
  - `courseDay` 空值或非正数写 `day-1`（不是 `day-all`）。

示例：`xxx-unit@day-3@collect_it`、`all@day-1@collect_it`。

### 2.2 类型 → 路由映射

文件：`lib/ui/ket/config/ket_game_flow_config.dart`

六阶主线相关条目：

| gameTypeName | route |
|---|---|
| `collect_it` | `/ket/collect-it-main` |
| `collect_it_game` | `/ket/collect-it` |
| `decode_it` | `/ket/decode-it` |
| `phrase_it` | `/ket/phrase-it` |
| `order_it` / `fill_it` | `/ket/order-it`（但 loading 分发时会改走 `/ket/friday`） |
| `sight_word_chant` | `/ket/sight-word-chant` |
| `game` | `/ket/friday` |

- 固定顺序：`ketGameTypeSequence = [collect_it, decode_it, phrase_it, order_it, sight_word_chant]`
- 下一个游戏类型：`nextKetGameTypeName(current)`。
- packId 解析：`resolveKetPackIdForGameType(packs, gameType)`
  优先按 `packs[i].gameTypeName` 匹配，未命中时按 `ketGameTypeSequence` 下标回落。

### 2.3 持久化与内存缓存

| 存储 | 作用 | 谁写入 | 谁读取 |
|---|---|---|---|
| **game-index 磁盘缓存** | 记录每个课次下有哪些 pack 以及它们的 `packId / version / hash / size / gameTypeName` | `/ket/resource-loader` 调 `preparePacks` 时写入 | 树桩页、collect-it-main、collect-it、`auto_fetch_pack` 分支 |
| **`<packDir>/content/pack.json` + 资源文件** | 单个包的实际场景数据（元素、规则、资源路径） | resource-loader / collect-it（按需下载）解压后落盘 | main 页、collect-it 页 |
| **`KetGamePackBindingStore`**（KV） | 业务键 → packId 的绑定（`ket_game_pack_binding_<cacheKey>`） | resource-loader、树桩页 | 树桩页 fallback |
| **`KetCollectItPackMemoryCache`**（内存） | `packId+version+hash → 解析好的 KetPrimitivePackData` | main / collect-it 加载成功后 | main / collect-it |
| **`KetCollectItProgressStore`**（KV） | main 场景每个 elementId 的完成态（按 main 的 packId 分键） | collect-it 通关时写入、main 的"清缓存"按钮清除 | main 页渲染 + collect-it 奖励判定 |
| **`GamePackRuntimeManager._routePackIdMap`**（内存） | 路由 path → packId 映射，仅保留最近一次 `refreshGameIndex` 的结果（覆盖写） | 树桩页调 `refreshGameIndex` 成功后由 `_buildRoutePackIdMap` 写入 | `app.dart` 多包游戏链路 fallback（六阶单包玩法不使用） |

## 3. 页面职责与数据流

### A. 资源加载页 `ket_resource_loader_page.dart`

**职责**：拉取并落盘资源包，写入业务绑定，跳转到树桩分发页。

关键行为：

1. 调 `GamePackRuntimeManager.instance.preparePacks(courseUnitId, courseDay, subLevel, levelId, theme, forceRefresh:false)`
   - 会请求 `GET /api/v1/runtime/game-index`
   - 按返回的 packs 逐个走 presign → 下载 → 校验 → 解压 → 写 manifest
2. `loadCachedIndex(...)` 读回刚写入的索引。
3. 对 `ketGameTypeSequence` 中每种类型用 `resolveKetPackIdForGameType` 挑出 packId，
   调 `KetGamePackBindingStore.savePackId(cacheKey, packId)` 写入绑定。
4. `context.pushReplacement('/ket/loading?game_type_name=...&courseUnitId=...&level_id=...&sub_level=...')`。
   **不透传 `course_day`**（下游不使用）。

失败：Toast "资源加载失败，请稍后重试" + 「重新加载」按钮。

### B. 树桩分发页 `ket_loading_page.dart`（路由 `/ket/loading`）

**职责**：用户可见的 6 树桩菜单 + 自动分发。

入参：`target, game_type_name, courseUnitId, course_day, auto_fetch_pack, sub_level, level_id, theme`。

> **`refreshGameIndex` 副作用**：树桩页在展示包列表前会调用
> `GamePackRuntimeManager.instance.refreshGameIndex(...)`。
> 成功后除写磁盘 game-index 缓存外，还会触发 `_buildRoutePackIdMap`，
> 将本次所有 pack 的 `gameTypeName → 路由 path → packId` 写入内存映射
> `_routePackIdMap`。该 map 供 `app.dart` 在多包游戏链路（listening /
> speaking / wrap-up / role-play / level-up 等）中，当后续子关卡 URL
> 未携带 `pack_id` 时用路由 path 回落查询（见第 9 节）。
> 六阶单包玩法（collect_it / decode_it 等）不使用此 map。

#### 模式 1：手动点击树桩

- `_stumpPacksFromIndex` 非空（即本地 game-index 有 packs）时，6 个树桩完全按
  `packs[0..5]` 绑定；否则用默认图。
- 点击后走 `_pushGameRoute` 的多级 packId 解析：
  1. 如果点的就是 `packs[i]`（`packFromApi`），直接用 `pack.packId`。
  2. 否则按 `resolveKetPackIdForGameType(packs, normalized)` 查当前过滤桶。
  3. 带 theme/level/subLevel 时，miss 后退回不带过滤条件再查。
  4. 再 miss 读 `KetGamePackBindingStore.getPackId(cacheKey)`。
  5. 若本地绑定 packId 和索引里的 `gameTypeName` 不一致，用索引覆盖（纠错）。
  6. 命中则回写绑定，`context.push(路由?pack_id=...&game_type_name=...&courseUnitId=...&level_id=...&sub_level=...)`。
- 特殊重定向：`game` / `order_it`（含 `fill_it` 别名）都走 `/ket/friday`。

#### 模式 2：`auto_fetch_pack=1`

用在 decode-it 等玩法通关后自动串联下一玩法：

1. 按 `game_type_name` 算路由。
2. 读本地绑定 + index → 取到 `GamePackInfo` → `preparePackByPackVersion(forceRefresh:false)` 做本地校验。
3. 校验失败就 `preparePacks(forceRefresh:false)` 重新刷索引并重新写绑定，再校验一次。
4. 成功：`context.go(路由?pack_id=...&game_type_name=...&courseUnitId=...)`。
5. 失败：Toast "未找到可用的下一个游戏资源"。

#### UI 其它

- 进度条 5s 从 0→100% 播完自动隐藏，纯视觉，不参与跳转。
- auto_fetch 期间有全屏半透明遮罩 + CircularProgressIndicator。

### C. Collect It 主场景 `ket_collect_it_main_page.dart`（路由 `/ket/collect-it-main`）

**职责**：加载 main 包，渲染主场景；用户点击元素触发 jump action 时，
把子包 packId 与 main 的 interactive 元素总数一起透传给 `/ket/collect-it`。

入参：`pack_id, game_type_name, courseUnitId, course_day`。

#### `_loadPackContent` 流水线

1. 仅用本地缓存索引：`GamePackRuntimeManager.loadCachedIndex(courseUnitId, courseDay)`。
2. `loadCachedPackInfoByPackId(widget.packId, courseUnitId, courseDay)` 拿 `GamePackInfo`；
   没有 packId 入参时退化为 `packs.first`。
3. `preparePackByPackVersion(forceRefresh:false)` 校验本地安装。
4. 内存缓存 peek；miss 则读 `pack.json` → `KetPrimitivePackAdapter.fromPackJson` → `remember`。
5. `KetCollectItProgressStore.getCompletedElementIds(packId)` 读取**完成态**并传给
   `KetPrimitiveRenderer` 做半透明渲染判定。
6. 统计 **interactive 元素** = `tap/drag/finish` 三类交互元素集合；**集合大小**会作为
   `interactiveCount` 注入 `_CollectItJumpExecutor`。
7. **不预热子包**：jump target 的下载/解析全部延迟到 collect-it 页自己完成。
8. 构造 `_CollectItJumpExecutor` 作为 runtime executor，注入：
   - `resolveSourceElementId`
   - `interactiveCount: _interactiveElementIds.length`
   - `isJumpLocked / setJumpLocked`（连点防抖）

#### 半透明渲染

`KetPrimitiveRenderer`（`lib/ui/ket/runtime/ket_primitive_renderer.dart`）：

- 对 interactive 集合里**未完成**的元素用 `inactiveOpacity=0.5` 渲染（半透明，
  提示"还可以点"）。
- **已完成**的元素恢复原不透明度，视觉上"点亮"。
- 这个判定每次 main 重建时都按 `_completedElementIds` 重新计算；
  collect-it 玩完 pop 回 main 时，`didPopNext` 会调用 `_refreshCompletedFromCache()`
  重新读 `KetCollectItProgressStore` → setState → 刚完成的洞立即变正常。

#### 右下角"清空缓存"按钮

调 `_clearCompletedCache()`：
- `KetCollectItProgressStore.clearCompletedElements(_activePackId)` 删除磁盘完成态；
- `setState` 清空 `_completedElementIds` → 所有 interactive 洞**恢复半透明**；
- 用于测试 / 调试，正式用户一般不会触发。

#### main → collect-it 跳转（`_CollectItJumpExecutor.execute`）

用户点一个主场景元素 → primitive runtime 触发该元素的 jump action → 走这个 executor：

1. 从 action 的 `params['target_pack_id']` / `params['targetPackId']` 取目标子包 id；
   空则 Warning log 直接 return（**不再兜底用 main 的 packId**）。
2. 从 `triggerSourceElementId` 或 `resolveSourceElementId(action)` 得到**本次点击的元素 id**
   （`sourceElementId`）。
3. **连点防抖**：
   - `isJumpLocked()==true` → 日志 `jump 已在进行，忽略重复触发` 并 return；
   - 否则 `setJumpLocked(true)` → `await context.push(uri)` → `finally { setJumpLocked(false); }`。
   - push 的 Future 只有 collect-it pop/销毁 后 resolve，确保 main 在这期间忽略所有重复触发。
4. 组装 URL：

   ```
   /ket/collect-it
     ?pack_id=<target 子包 id>
     &source_pack_id=<main 的 packId>         ← 用于写完成态
     &source_element_id=<本次点击的元素 id>
     &main_interactive_count=<main 的 interactive 总数>
     &game_type_name=<nextGameTypeName>
     &courseUnitId=<id>
   ```

   说明：
   - `source_pack_id` + `source_element_id` 是 collect-it 写完成态的唯一信息来源。
   - `main_interactive_count` 是 collect-it 判定"是否全通关、是否弹庆祝动画"的依据，
     替代了旧字段 `should_reward`（已从全链路删除）。
   - 不透传 `course_day`（collect-it 不用）。

#### 其它

- BGM `KET_Collect _Main.mp3` 循环；`didPushNext` 停，`didPopNext` resume（从 collect-it pop 回来可听回 main 的 BGM）。
- RouteAware：`didPopNext` 和 `didChangeDependencies` 里都会刷新完成态。
- 生命周期内 `_jumpInFlight` 默认 false，只有在 jump → push 期间是 true。

### D. Collect It 游戏页 `ket_collect_it_page.dart`（路由 `/ket/collect-it`）

**职责**：按 `pack_id` 独立完成资源包准备 → 玩法交互 → 写完成态 → 按是否全通关决定回路。

入参：
`pack_id, source_pack_id, source_element_id, main_interactive_count, game_type_name, courseUnitId, course_day, isLast`。

#### D.1 `_loadPackData` 四级 fallback

```
A. widget.packId 为空 → 本地错误 "缺少 pack_id"

B. loadCachedPackInfoByPackId(widget.packId, courseUnitId: widget.courseUnitId)
   → 本地 game-index 中按 packId 查 GamePackInfo(version/hash/size)
   null → 本地错误 "未找到资源包索引，请先进入资源加载页"

C. KetCollectItPackMemoryCache.peek(packId, version, hash)
   命中 → 直接使用（最快路径）

D. 本地磁盘完整性判断
   GamePackEntryUtil.hasLocalPackResources(cachedPack) == true
   → 读 pack.json → 解析 → remember → 使用

E. 仍未拿到 → presign 下载
   GamePackEntryUtil.ensurePackReady(cachedPack, dataSource: useLocalWhenReady)
     内部：POST /api/v1/runtime/game-pack/presign
          → 下载 zip → _verifyHash → 解压到 content/ → 写 manifest.json
   成功 → 再次读 pack.json → 解析 → remember → 使用
   失败（ApiException）→ _presignErrorMessage = e.message，展示错误卡片
   失败（其它异常）     → _presignErrorMessage = e.toString()，展示错误卡片

F. 成功路径：_hydrateFromParsedPack + precacheImage(background)
```

**关键点**：

- **完全不依赖 main 的预热**；即使 main 页跳过了任何 jump target，
  collect-it 页自己也能按 packId 自愈。
- version/hash 来自本地 game-index 缓存（B 步），所以 resource-loader 必须
  至少成功跑过一次。
- presign 调用一次后会把 zip 下载到
  `Documents/game_packs/<packId>/<version>/` 并 `manifest.json` 标记已装；
  下次进入走 D 步命中，不会重复下载。
- 错误文案**原文展示**（来自后端 `ApiException.message`），不做业务分类改写。

#### D.2 加载与错误 UI

- **卡通 Loading 场景**（`_CartoonLoadingScene`）：
  背景 `assets/ket/ket_loading_background.png` + 中央角色
  `get_stratrd.png`，1200ms 周期"呼吸 + 左右摇摆"动画，下方 "加载中..." 文案。
  不显示进度条。
- **错误卡片**（`_CollectItErrorScene`）：
  - presign/下载失败：展示原文 + 「重试」+「返回」并列；
    重试通过 `_retryLocked` 瞬时锁防连点。
  - 本地配置类错误（`_packErrorMessage`）：只展示「返回」。
  - 返回走 `Navigator.of(context).maybePop()`。

#### D.3 玩法完成判定

所有 drag 元素完成 → `_runFinishActions`：

1. `_revealFinishElements` 淡入 + 上移 10px，700ms 展示 finish 元素。
2. 执行所有 onFinish 的非 snap actions。
3. **`_markMainElementCompleted`**（返回 `bool _wasNewlyCompleted`）：
   - `progressPackId = source_pack_id.isNotEmpty ? source_pack_id : packId`。
   - 读现有完成集 → 比较 `source_element_id` 是否已存在 → `wasNew`。
   - `KetCollectItProgressStore.markElementCompleted(packId=progressPackId, elementId=source_element_id)`。
   - 返回 `wasNew`；赋值给 `_wasNewlyCompleted` 字段。
4. 播卡片 Lottie（动态替换 `image_2` 为 `scene.rewardUrl`）；**每次玩完都播**，
   这是默认流程的一部分，与奖励判定解耦。

#### D.4 卡片动画播完 → `_decideAfterCardAnimation`

```
completedCount = KetCollectItProgressStore.getCompletedElementIds(progressPackId).length
reachedAll     = mainInteractiveCount > 0 && completedCount >= mainInteractiveCount
playCelebration = _wasNewlyCompleted && reachedAll

if (playCelebration) {
    _triggerFinishCelebration()           // 弹 GameFinishCelebrationOverlay
    // overlay onDismissed 回调 → _goToCourseGamesTransition()
} else {
    await Future.delayed(300 ms)
    _popBackToMain()                       // 回 main 继续点下一个洞
}
```

三种场景对应：

| 用户动作 | `_wasNewlyCompleted` | `reachedAll` | 结果 |
|---|---|---|---|
| 点了一个新洞且不是最后一个 | true | false | 300ms → pop 回 main（完成态刷新） |
| 点了最后一个没完成的洞 | true | true | 弹庆祝 → `go /ket/loading` 进树桩选下一玩法 |
| 重玩已完成的洞 | false | 视情况 | 300ms → pop 回 main（**不重弹庆祝**） |

#### D.5 返回路径

- **`_popBackToMain`**：优先 `Navigator.of(context).pop()` 回 main（触发 main 的
  `didPopNext` → 刷新完成态 → 刚完成的洞变正常）；
  栈上无可 pop（如深链接直接进入）则兜底 `_goToCourseGamesTransition`。
- **`_goToCourseGamesTransition`**：`context.go('/ket/loading?game_type_name=<current>&courseUnitId=<id>')`。
  `go` 会清栈，main 与 collect-it 都被替换。
  - 庆祝 overlay 关闭、**深链接兜底**、**顶部设置条右侧返回按钮**都走这条路径。
  - 左侧 Home 按钮直接 `context.go('/ket/kingsley')`，独立语义。

#### D.6 完成态 key 的选择

`_markMainElementCompleted` 和 `_decideAfterCardAnimation` 都用
`sourcePackId.isNotEmpty ? sourcePackId : packId` 作为 progressStore 的 key：

- 正常场景：`sourcePackId` 是 main 的 packId，**完成态写在 main 包下**，这样 main
  渲染器一次性就能把所有洞标记到位。
- 异常场景：`sourcePackId` 缺失（老 URL、深链接）→ 退化为用子包 packId；只能保证
  自洽，不能和 main 同步。

## 4. main ↔ collect-it 闭环详细时序

```
┌────────────── /ket/collect-it-main ──────────────┐
```

## 5. 路由参数约定

文件：`lib/app.dart`

| 路由 | 入参 |
|---|---|
| `/ket/resource-loader` | `courseUnitId, course_day│courseDay, game_type_name, sub_level, level_id, theme` |
| `/ket/loading` | `target, game_type_name, courseUnitId, course_day│courseDay, auto_fetch_pack, sub_level, level_id, theme` |
| `/ket/collect-it-main` | `pack_id│packId, game_type_name, courseUnitId, course_day│courseDay` |
| `/ket/collect-it` | `pack_id, source_pack_id, source_element_id, main_interactive_count, game_type_name, courseUnitId, course_day│courseDay, isLast` |

> **`course_day` 目前只在 resource-loader 入口被接收用于写缓存键和请求过滤**。
> 后续页面跳转 URL 里不再透传，所以 loading/main/collect-it 拿到的
> `courseDay == null`，它们的 `buildKetGamePackCacheKey` 都会用默认的 `day-1`。
> 这是有意选择；若以后需要按 day 分桶，需同时在多个 URL 补齐。

> **`should_reward` 已从全链路删除**（main、collect-it、路由解析），
> 由 `main_interactive_count` + progressStore 实时计算取代。

## 6. 闭环流程（当前实现）

1. 进入 `resource-loader`：请求 game-index、下载解压、写绑定。
2. `pushReplacement` 到 `/ket/loading`。
3. `loading`：
   - 手动点树桩 → 按 game-index + 绑定解析 packId，`push` 到具体玩法路由。
   - 或 `auto_fetch_pack=1` 自动分发（decode-it 通关后使用）。
4. `collect-it-main`：本地缓存里读 main 包；按 interactive 集合渲染半透明洞。
5. 用户点一个洞 → `push /ket/collect-it`，URL 带 main 的 interactive 总数。
6. `collect-it`：内存→磁盘→presign 下载；成功后玩法进行 → 写完成态 → 卡片 Lottie。
7. 卡片结束后：
   - 全通关（新完成 && 总数达标）→ 庆祝动画 → `go /ket/loading` 进树桩页。
   - 未全通关 → `pop` 回 main，刚完成的洞变正常。
8. 回到树桩页后，用户手动选下一玩法，或 decode-it 等通过 `auto_fetch_pack=1` 自动串联。

## 7. 关键收益

- **main 首帧更轻**：删除了 `prefetchAllJumpTargetsForMainScene` 预热，
  不再一次性解析所有 jump target。
- **collect-it 能自愈**：即使 resource-loader 未能覆盖到某个子包，
  collect-it 也能按 packId 走 presign + 本地缓存补拉，不再卡"未命中缓存"。
- **面向用户的错误可见**：presign 失败直接展示后端原文，用户可「重试」或「返回」。
- **连点防抖**：main 的 jump executor 通过 `_jumpInFlight` 锁住，
  快速双击不会出现多层 collect-it 叠栈。
- **奖励判定正确**：基于 progressStore 实时计算，不再依赖 main 的内存快照；
  重玩已完成的洞不会重复弹庆祝；`main_interactive_count=0` 的老 URL 自然兼容。
- **多洞体验连贯**：未全通关时 `pop` 回 main 而不是 `go` 清栈到树桩，用户无需
  重新点一次"Collect It"树桩，流畅完成主场景全部洞。

## 8. 未涵盖的玩法

- `decode_it / phrase_it / order_it / sight_word_chant / friday`
  仍走"树桩页 → 玩法页"单包模式，资源包都在 resource-loader 阶段预拉。
- 这些玩法通关后如需自动串行，通过 `auto_fetch_pack=1` 回到 loading 分发。
- 没有类似 collect-it 的"main + 子包"两层结构，也没有用 progressStore 的进度语义。

## 9. 多包游戏路由机制（`sec_pack_id` 移除后的新方案）

### 9.1 背景

KET 中存在若干"多包顺序游戏"：一次游戏由 2～3 个 pack 组成，玩完一包自动进入下一包。
例如：`listening_lets_listen`（Q1 → Q2）、`ready_set_talk`（main → sec）、
`wrap_up_reading / speaking / listening`（one → two）、`role_play`（多阶段）、
`level_up`（one → two → three）。

旧方案：第一关在跳转 URL 中携带 `sec_pack_id` / `tip_pack_id` / `two_pack_id` 等
伴生参数显式传给下一关。该方案依赖树桩页在跳转 URL 里提前填好这些参数，
而树桩页实际并不填写，导致伴生参数恒为空，多包链路断开。

### 9.2 新方案：路由 path → packId 内存映射

核心：树桩页调 `refreshGameIndex` 成功后，`_buildRoutePackIdMap` 将本次所有 pack
的 `gameTypeName` 经 `ketRoutePathForGameType` 转为路由 path，以 path 为键写入
`_routePackIdMap`（内存，覆盖写，仅保留最近一次结果）。

```
gameTypeName → ketRoutePathForGameType() → "/ket/xxx" → map["/ket/xxx"] = packId
```

**`app.dart` fallback 逻辑**：多包链路中第 2+ 段路由的 builder 先尝试从 URL query
取 `pack_id`，无值时调 `GamePackRuntimeManager.instance.packIdForRoute(path)` 查 map：

```dart
final String? packId = state.uri.queryParameters['pack_id']
    ?? GamePackRuntimeManager.instance.packIdForRoute('/ket/目标路由');
```

涉及的路由：`/ket/listening_lets_listen_two`、`/ket/speaking/ready-set-talk-sec`、
`/ket/speaking/role-play-one-choose`、`/ket/speaking/role-play-one-second`、
`/ket/speaking/role-play-two`、`/ket/speaking-level-up-two`、
`/ket/speaking-level-up-three`、`/ket/wrap-up-reading-two`、
`/ket/wrap-up-speaking-two`、`/ket/wrap-up-listening-two`。

### 9.3 伴生参数清理

以下参数已从代码中全部删除（它们均为废码，树桩页从未传入）：

- `sec_pack_id` / `secPackId`（listening_lets_listen、ready_set_talk 的第二包 ID）
- `tip_pack_id`、`choose_pack_id`、`second_pack_id`、`two_pack_id`、
  `three_pack_id`（role_play / level_up 的伴生包 ID）

同时删除了各游戏页内部通过 `KetGamePackBindingStore.packIdForRole` 和
game-index 四级回落解析多包 packId 的逻辑（wrap_up_reading / speaking / listening）。

### 9.4 约束与注意事项

- `_routePackIdMap` 仅保留最近一次 `refreshGameIndex` 的结果。若用户
  在不同课次间切换，map 会被覆盖为新课次的数据，旧课次的多包链路应在
  同一次进入树桩页后连续完成，不跨课次混用。
- 若 `refreshGameIndex` 失败（网络异常），map 保持上一次的结果或为空；
  此时后续多包子关卡若 URL 同样无 `pack_id`，则进入加载失败流程。
- 六阶单包玩法（collect_it / decode_it / phrase_it / order_it / sight_word_chant）
  不受影响，其 packId 仍由树桩页在跳转 URL 中显式携带。
